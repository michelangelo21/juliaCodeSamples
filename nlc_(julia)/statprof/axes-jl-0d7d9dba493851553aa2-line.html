1 """
2 ro = IdOffsetRange(r::AbstractUnitRange, offset=0)
3
4 Construct an "identity offset range". Numerically, `collect(ro) == collect(r) .+ offset`,
5 with the additional property that `axes(ro, 1) = axes(r, 1) .+ offset`.
6 When `r` starts at 1, then `ro[i] == i` and even `ro[ro] == ro`,
7 i.e., it's the "identity," which is the origin of the "Id" in `IdOffsetRange`.
8
9 # Examples
10
11 The most common case is shifting a range that starts at 1 (either `1:n` or `Base.OneTo(n)`):
12 ```jldoctest; setup=:(import OffsetArrays)
13 julia> ro = OffsetArrays.IdOffsetRange(1:3, -2)
14 OffsetArrays.IdOffsetRange(-1:1)
15
16 julia> axes(ro, 1)
17 OffsetArrays.IdOffsetRange(-1:1)
18
19 julia> ro[-1]
20 -1
21
22 julia> ro[3]
23 ERROR: BoundsError: attempt to access 3-element UnitRange{$Int} at index [5]
24 ```
25
26 If the range doesn't start at 1, the values may be different from the indices:
27 ```jldoctest; setup=:(import OffsetArrays)
28 julia> ro = OffsetArrays.IdOffsetRange(11:13, -2)
29 OffsetArrays.IdOffsetRange(9:11)
30
31 julia> axes(ro, 1) # 11:13 is indexed by 1:3, and the offset is also applied to the axes
32 OffsetArrays.IdOffsetRange(-1:1)
33
34 julia> ro[-1]
35 9
36
37 julia> ro[3]
38 ERROR: BoundsError: attempt to access 3-element UnitRange{$Int} at index [5]
39 ```

41 # Extended help
42
43 Construction/coercion preserves the (shifted) values of the input range, but may modify
44 the indexes if required by the specified types. For example,
45
46 r = OffsetArrays.IdOffsetRange{Int,UnitRange{Int}}(3:4)
47
48 has `r[1] == 3` and `r[2] == 4`, whereas
49
50 r = OffsetArrays.IdOffsetRange{Int,Base.OneTo{Int}}(3:4)
51
52 has `r[3] == 3` and `r[4] == 4`, and `r[1]` would throw a `BoundsError`.
53 In this latter case, a shift in the axes was needed because `Base.OneTo` ranges
54 must start with value 1.
55
56 !!! warning
57
58 In the future, *conversion* will preserve both the values and
59 the indices, throwing an error when this is not achievable. For instance,
60
61 r = convert(OffsetArrays.IdOffsetRange{Int,UnitRange{Int}}, 3:4)
62
63 has `r[1] == 3` and `r[2] == 4` and would satisfy `r == 3:4`, whereas
64
65 ```julia
66 julia> convert(OffsetArrays.IdOffsetRange{Int,Base.OneTo{Int}}, 3:4) # future behavior, not present behavior
67 ERROR: ArgumentError: first element must be 1, got 3
68 ```
69
70 where the error will arise because the result could not have the same axes as the input.
71
72 An important corollary is that `typeof(r1)(r2)` and `oftype(r1, r2)` will behave differently:
73 the first coerces `r2` to be of the type of `r1`, whereas the second converts.
74 Developers are urged to future-proof their code by choosing the behavior appropriate for each usage.
