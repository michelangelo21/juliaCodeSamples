16 """
17 IndexLinear()
18
19 Subtype of [`IndexStyle`](@ref) used to describe arrays which
20 are optimally indexed by one linear index.
21
22 A linear indexing style uses one integer index to describe the position in the array
23 (even if it's a multidimensional array) and column-major
24 ordering is used to efficiently access the elements. This means that
25 requesting [`eachindex`](@ref) from an array that is `IndexLinear` will return
26 a simple one-dimensional range, even if it is multidimensional.
27
28 A custom array that reports its `IndexStyle` as `IndexLinear` only needs
29 to implement indexing (and indexed assignment) with a single `Int` index;
30 all other indexing expressions — including multidimensional accesses — will
31 be recomputed to the linear index. For example, if `A` were a `2×3` custom
32 matrix with linear indexing, and we referenced `A[1, 3]`, this would be
33 recomputed to the equivalent linear index and call `A[5]` since `2*1 + 3 = 5`.
34
35 See also [`IndexCartesian`](@ref).
36 """
37 struct IndexLinear <: IndexStyle end

39 """
40 IndexCartesian()
41
42 Subtype of [`IndexStyle`](@ref) used to describe arrays which
43 are optimally indexed by a Cartesian index. This is the default
44 for new custom [`AbstractArray`](@ref) subtypes.
45
46 A Cartesian indexing style uses multiple integer indices to describe the position in
47 a multidimensional array, with exactly one index per dimension. This means that
48 requesting [`eachindex`](@ref) from an array that is `IndexCartesian` will return
49 a range of [`CartesianIndices`](@ref).
50
51 A `N`-dimensional custom array that reports its `IndexStyle` as `IndexCartesian` needs
52 to implement indexing (and indexed assignment) with exactly `N` `Int` indices;
53 all other indexing expressions — including linear indexing — will
54 be recomputed to the equivalent Cartesian location. For example, if `A` were a `2×3` custom
55 matrix with cartesian indexing, and we referenced `A[5]`, this would be
56 recomputed to the equivalent Cartesian index and call `A[1, 3]` since `5 = 2*1 + 3`.
57
58 It is significantly more expensive to compute Cartesian indices from a linear index than it is
59 to go the other way. The former operation requires division — a very costly operation — whereas
60 the latter only uses multiplication and addition and is essentially free. This asymmetry means it
61 is far more costly to use linear indexing with an `IndexCartesian` array than it is to use
62 Cartesian indexing with an `IndexLinear` array.
63
64 See also [`IndexLinear`](@ref).
65 """

197 # check for valid sizes in A[I...] = X where X <: AbstractArray
198 # we want to allow dimensions that are equal up to permutation, but only
199 # for permutations that leave array elements in the same linear order.
200 # those are the permutations that preserve the order of the non-singleton
201 # dimensions.
202 function setindex_shape_check(X::AbstractArray, I::Integer...)
203 li = ndims(X)
204 lj = length(I)
205 i = j = 1
206 while true
207 ii = length(axes(X,i))
208 jj = I[j]
209 if i == li || j == lj
210 while i < li
211 i += 1
212 ii *= length(axes(X,i))
213 end
214 while j < lj
215 j += 1
216 jj *= I[j]
217 end
218 if ii != jj
219 throw_setindex_mismatch(X, I)
220 end
221 return
222 end
223 if ii == jj
224 i += 1
225 j += 1
226 elseif ii == 1
227 i += 1
228 elseif jj == 1
229 j += 1
230 else
231 throw_setindex_mismatch(X, I)
232 end
233 end
234 end
